Overview

Overview of Testing Environment

For my final project in the MSR program, I am creating a dynamic testing environment that uses multiple robot arms to present objects and various stimuli to rodents. The goal is to measure rodents' behavioral responses and track their whisker movements as they sense the objects and interact with the arms.

ROS Implementation and Hardware

I am using two WidowX 200 arms from Trossen Robotics. The Dynamixel motors on the robot arms can be sent position, velocity, and effort values over 3 Pin TTL enabled by U2D2 USB communication converters from Robotis.

PC to Dynamixel Connection (credit: Trossen Robotics)

The arms are controlled within a ROS framework that enables researchers to easily program behavioral experiments. I am using MoveIt to generate robot arm motion plans while avoiding collisions with both rodents and testing environment fixtures. All programming for this project is being completed in C++, but non-technical users will be able to specify trajectory sequences by manually manipulating the arms and saving each pose through a custom GUI created in Librviz. Users will be able to place path constraints and virtual structures in the planning scene through this interface. In the future, waypoints may also be specified through a Python API, or loaded yaml config file.

Feature Demos:

1 - MoveIt Planning Interface

Users can specify waypoints to one or more arms in the testing environment by using the MoveGroupInterface class. A URDF file combining both arms in the testing environment was written to enable collision checking without external sensors. A relative position and orientation offset between the two arms can be passed in as a set of ROS parameters at runtime, enabling users to seamlessly change the positions of the arms for different experimental setups while still having collision checking. A demo showcasing both arms moving to a set of user-specified waypoints can be seen below.

2 - Visualization and Virtual Objects

Each arm in the testing environment has its own control node that is responsible for interfacing with the actual hardware - accepting joint and end-effector commands through Dynamixel's Protocol 2.0. Message_filters were used to read current joint states from each arm and send them to a combined /testing_environment/joint_states topic that enables users to visualize the environment in real-time through Rviz. I used MoveIt's PlanningScene class to add virtual collision objects to the environment, such as the perch the rodents will enter the environment from when they are presented with objects. Ground planes and other types of constraints can also be specified through this class. A side-by-side demo of a real-world motion plan with an Rviz environment and virtual perch object can be seen below.

3 - Manual Waypoint Setting

Since the primary users of this environment may not have much experience with ROS or robotic hardware, I created a set of nodes that enables users to specify waypoints by manually manipulating the arms. This feature will be particularly useful for specifying robot arm poses that have very tight clearances, or for quickly adding a new pose to an experiment. A demo of the manual waypoints feature can be seen below.

Behavioral Decision Process

During experimental trials, a rodent will enter the testing environment and be presented with at least two objects. These objects may differ in shape, color, or texture. If the rodent appears to "choose" one of the objects (by sensing it with its whiskers or touching it with its paws), the arm holding that object will replace it with a different object as part of a user-specified sequence.

Model of presented object sequence based on rodent's decisions

A new object may be presented to the rodent at a different position within the rodent's whisker receptive field, or be placed on another side of the environment. A stepper motor with multiple objects may be attached to the end effector to enable quick object swapping. Other future work includes force control on larger objects, and motion planning integration with IR or RGBD cameras.

Background

The basic idea behind a generative adversarial network is to pit two different machine learning models against each other in a min-max game. Both networks - one called a "generator" and one called a "discriminator", attempt to minimize their loss functions at the behest of the other model. The goal of the generator is to create data that matches the variation and distribution of some target data set. In theory, this synthetic data could be used to train other models with the same level of success as "real" data if it truly matches a target data set's variety and distribution. On the other hand, the goal of the discriminator is to be able to discern "real" data from "fake" data created by the generator.

Block diagram of a basic GAN

The ideal training outcome is to have both models reach a sort of Nash equilibrium with each other. If the discriminator becomes too good at discerning real data from fake data early on in the training process, your generator may not be able to create progressively more "realistic" data with each training run. On the other hand, your generator can overpower your discriminator with garbage data that does not resemble the target data set.

2-D Architectures

I implemented a Deep Convolutional architecture (DCGAN) to generate data that matched the variety and distribution of MNIST and MNIST FASHION datasets. I also implemented a shallow, two-layer architecture to generate plot points that matched the distribution of an unknown function in a 2-D coordinate space.
